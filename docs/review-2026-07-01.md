## Critical (must fix)

### C1. `effectCapRatio: NaN` silently disables every timed event

**File:** `src/main/java/org/tupi/randomchaos/config/ChaosConfig.java:67`

```java
if (loaded.effectCapRatio <= 0.0 || loaded.effectCapRatio > 1.0) { ... }
```

`Double.NaN <= 0.0` and `Double.NaN > 1.0` are both `false` (IEEE 754), so a
hand-edited `"effectCapRatio": NaN` passes validation. Gson 2.13+ parses
unquoted `NaN` in lenient mode. `Math.round(intervalTicks * NaN) = 0`, so every
`defaultDurationTicks() > 0` event becomes instant. The HUD never shows
"Now: <event> → <victim>". The mod appears broken.

**Impact:** Security — a config edit or crash can silently disable the mod's
central mechanic.

**Fix:**
```java
if (!Double.isFinite(loaded.effectCapRatio) || loaded.effectCapRatio <= 0.0 || loaded.effectCapRatio > 1.0) {
    RandomChaosMod.LOGGER.warn("randomchaos.json: effectCapRatio must be in (0.0, 1.0], clamping to 0.7");
    loaded.effectCapRatio = 0.7;
}
```

### C2. `intervalSeconds` integer overflow wraps next-event tick to negative

**File:** `src/main/java/org/tupi/randomchaos/config/ChaosConfig.java:63-65, 79-81`

```java
if (loaded.intervalSeconds <= 0) { ... clamp to 120 ... }         // no upper bound
public int intervalTicks() { return intervalSeconds * 20; }        // int multiplication
```

`intervalSeconds = 120_000_000` passes `> 0` and produces `ticks = -1_894_967_296`
(confirmed). Then `state.nextEventTick = now + negative → past`, so the scheduler
fires an event every tick — DoSing the server with zombie spawns and S2C
broadcasts.

**Impact:** Security — a config edit causes server spam, potential crash.

**Fix:**
```java
private static final int MAX_INTERVAL_SECONDS = 86_400; // 1 day
if (loaded.intervalSeconds <= 0 || loaded.intervalSeconds > MAX_INTERVAL_SECONDS) {
    RandomChaosMod.LOGGER.warn("randomchaos.json: intervalSeconds must be in (0, {}], clamping", MAX_INTERVAL_SECONDS);
    loaded.intervalSeconds = 120;
}
```
Consider also returning `long` from `intervalTicks()`.

### C3. `ClientChaosState` torn read across fields → HUD displays wrong time

**File:** `src/client/java/org/tupi/randomchaos/client/net/ClientChaosState.java:8-15`
**File:** `src/client/java/org/tupi/randomchaos/client/hud/ChaosHudOverlay.java:25`

Each field is `volatile`, but the receiver writes 8 fields sequentially and the
HUD reads them non-atomically:

```java
long nowServer = s.serverTick + (mc.level.getGameTime() - s.clientGameTimeAtReceive);
```

If the network thread updates `serverTick` _between_ the HUD reading `serverTick`
and `clientGameTimeAtReceive`, the result is `2000 + (gameTime - 500)` — wildly
wrong. Countdowns jump forward/backward by an entire interval.

**Impact:** Concurrency — visible HUD glitch, timer desync.

**Fix:** Publish an immutable snapshot over a single `volatile` reference:
```java
public record Snapshot(long serverTick, long challengeStartTick, ..., long clientGameTimeAtReceive) {}
private volatile Snapshot snapshot = new Snapshot(0, 0, ...);
public void update(Snapshot s) { snapshot = s; }
public Snapshot snapshot() { return snapshot; }
```
The HUD reads `INSTANCE.snapshot()` once per frame.

### C4. No-op mixins declared `required: true` will hard-crash on MC version bumps

**Files:**
- `src/main/java/org/tupi/randomchaos/mixin/RandomChaosMixin.java`
- `src/client/java/org/tupi/randomchaos/client/mixin/RandomChaosClientMixin.java`
- `src/main/resources/randomchaos.mixins.json`
- `src/client/resources/randomchaos.client.mixins.json`

Both `@Inject` methods have empty bodies. They inject into
`MinecraftServer.loadLevel()` and `Minecraft.run()` with `"required": true`
and `"defaultRequire": 1`. If either target is renamed in a future MC version,
the game fails to start — for zero benefit.

**Impact:** Build fragility — will silently break on update.

**Fix:** Delete both mixin classes and their `"mixins"` entries in both
`*.mixins.json` files and the `"mixins"` block in `fabric.mod.json`. If
scaffolding is desired, set `"required": false`.

## Important (should fix)

### I1. `ChaosState` exposes 8 public mutable fields — easy to mutate without `setDirty()`

**File:** `src/main/java/org/tupi/randomchaos/state/ChaosState.java:49-56`

Every field is `public long/String/UUID/int`. Any future contributor can write
`state.challengeStartTick = 0` and forget `setDirty()`, losing the change on
crash. `lastVictimUuid` and `consecutivePicks` are scheduler-internal state
leaked into the public SavedData API.

**Fix:** Make fields `private`, expose via getters, and provide mutation methods
that call `setDirty()`. At minimum narrow `lastVictimUuid`/`consecutivePicks`
to package-private.

### I2. Duplicate payload to the joining player

**File:** `src/main/java/org/tupi/randomchaos/lifecycle/ChaosLifecycle.java:19-25`

```java
onPlayerJoin(server);                                       // broadcasts to ALL
if (player != null) {
    ChaosScheduler.sendTo(player, ChaosState.get(...), ...); // sends AGAIN to joiner
}
```
The joiner is already in `getPlayerList().getPlayers()` when `JOIN` fires.

**Fix:** Delete lines 22-24.

### I3. HUD has no guard for "no payload received yet"

**File:** `src/client/java/org/tupi/randomchaos/client/hud/ChaosHudOverlay.java:25-28`

If the first payload arrives before `Minecraft.getInstance().level` is set,
`clientGameTimeAtReceive` stays 0. A later payload updates `serverTick` but
`clientGameTimeAtReceive` is stale, producing an inflated `nowServer`.

**Fix:** Detect "no payload yet" via `s.serverTick == 0 && s.challengeStartTick == 0`.
Combine with C3's immutable snapshot.

### I4. `ChaosStateManager` is dead code

**File:** `src/main/java/org/tupi/randomchaos/state/ChaosStateManager.java` (12 lines)

Delegates `get(server)` to `ChaosState.get(server)`. Never called. Delete.

### I5. `ChaosScheduler` mixes scheduling with networking and payload construction

**File:** `src/main/java/org/tupi/randomchaos/scheduler/ChaosScheduler.java:146-164`

`broadcast()`, `sendTo()`, `toPayload()` belong in `ChaosNetworking`.
Violates SRP.

**Fix:** Move to `ChaosNetworking`. Scheduler calls
`ChaosNetworking.broadcast(server, state.toPayload(serverTick))`.

### I6. `ChaosEventRegistry` read-side asymmetry

**File:** `src/main/java/org/tupi/randomchaos/event/ChaosEventRegistry.java:19-44`

`register()` is `synchronized`, but `pickRandom()`, `get()`, `size()`, `all()`
read with no lock. Safe today (single-writer at init), fragile if late
registration is ever added.

**Fix:** Document "single-writer at init, read-only after" or switch to
`ConcurrentHashMap` + `CopyOnWriteArrayList`.

### I7. `SpawnZombieEvent` spawns at `victim.getY()` with no safe-spawn check

**File:** `src/main/java/org/tupi/randomchaos/events/SpawnZombieEvent.java:29-41`

Underground or over-a-cliff: zombie spawns inside a block or mid-air.

**Fix:** Use `level.getHeightmapPos(MOTION_BLOCKING_NO_LEAVES, ...)` and verify
`level.noCollision(zombie.getBoundingBox())`. Retry radially if blocked.

### I8. Magic numbers in the HUD

**File:** `src/client/java/org/tupi/randomchaos/client/hud/ChaosHudOverlay.java:52-75`

Colors `0x55FF55`, `0x808080`, `0xFFFFFF`, `0xFFAA00`, `0x55FFFF`, margin `4`,
line height `11` — all inline.

**Fix:** Extract named constants.

### I9. Self-test never exercises the 16-rejection fallback

**File:** `src/main/java/org/tupi/randomchaos/ChaosSelfTest.java:70-108`

With 2 players, probability of 16 consecutive rejections in 500 iterations is
~0.0015%. The deterministic fallback (`ChaosScheduler.java:119-124`) is
untested.

**Fix:** Add a test that forces the fallback by seeding an RNG that always
returns the blocked player.

### I10. `clampDuration` edge cases untested

**File:** `src/main/java/org/tupi/randomchaos/ChaosSelfTest.java:46-51`

All 5 cases use `intervalTicks = 2400`. Zero-interval and negative-duration
paths are untested.

**Fix:** Add `clampDuration(100, 0, 0.7) == 0` and a negative case.

### I11. `event.apply` failure still updates `nextEventTick` and HUD state

**File:** `src/main/java/org/tupi/randomchaos/scheduler/ChaosScheduler.java:77-101`

If `apply` throws, the catch continues to write `currentEventId`,
`currentEffectExpiryTick`, etc. For timed events the HUD displays a duration
the world never experienced.

**Fix:** Skip the state-update block inside the catch, or add a comment
documenting the intentional design (nextEventTick advances either way to
prevent retry loops, and the HUD will correct on the next broadcast).

## What's done well

- **Spec adherence is near-perfect.** All sections 1–11 of `spec.md` are
  implemented. No scope creep. The 5-agent parallel orchestration produced a
  coherent codebase with zero signature drift.
- **Defensive event application.** `event.apply(victim)` is wrapped in
  `try/catch (Throwable)` — a buggy event never crashes the tick loop.
- **Source-set split is correct.** `src/main` for common+server, `src/client`
  for client-only. Mixin scoping in `fabric.mod.json` uses
  `"environment": "client"` correctly.
- **Config robustness.** Empty file → defaults. Malformed JSON → defaults.
  Locked directory → graceful log. Only `NaN` and integer overflow slip through
  (C1, C2).
- **`ChaosEvent` interface is clean.** 4 methods, easy to extend. Registry
  throws on duplicate IDs and empty picks. One new class + one line in
  `onInitialize()` per event.
- **`ChaosStatePayload` uses a `record` with `StreamCodec.composite`** —
  idiomatic vanilla 1.20.3+ style. The nullable-UUID codec is a nice touch.
- **`ChaosSelfTest` is a strong `@GameTest` substitute** given the 26.1.2
  structure-template requirement. 10 checks, throws on failure, prevents
  broken pure helpers from reaching production.
- **Mojang mappings only — no Yarn leakage.** All imports match the package
  layout documented in `AGENTS.md`.
